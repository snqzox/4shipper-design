## 2. Naming Consistency

### 2a. Case inconsistency within the `Route Entry` family

Three separate sets exist for what should be one component:

| Existing set name | Variant count |
|---|---|
| `Route Entry` | 11 |
| `Route entry` | 6 |
| `Route Entry 2` | 4 |

`Route entry` (lowercase e) differs only in capitalisation from `Route Entry`. `Route Entry 2` was probably a scratchpad copy that was never merged back. Any component referencing these by name will silently resolve to different things depending on casing.

**Recommended standard:** Consolidate into `Route Entry` (title case). Merge the 6 variants from `Route entry` in; investigate whether `Route Entry 2` adds unique states or can be removed.

### 2b. Singular vs. plural mismatch in the `Status/*` family

| Existing set name | Variant count |
|---|---|
| `Status/Transports` (plural) | 4 |
| `Status/Requests` (plural) | 4 |
| `Status/Contract` (singular) | 4 |
| `Status/Tender` (singular) | 5 |

`Transports` and `Requests` are plural; `Contract` and `Tender` are singular. Pick one convention and apply it across all sub-sets.

**Recommended standard:** Singular throughout — `Status/Transport`, `Status/Request`, `Status/Contract`, `Status/Tender`.

### 2c. `Carrier Selection` vs. `Carrier Selection/Entry` — undocumented relationship

| Set | Count |
|---|---|
| `Carrier Selection` | 4 |
| `Carrier Selection/Entry` | 2 |

`Carrier Selection` contains the expanded/collapsed card variants; `Carrier Selection/Entry` appears to be a sub-row item used inside the card. If that is the intent, the slash grouping is correct — but it is currently undocumented so consumers cannot tell when to use each.

**Recommended action:** Add descriptions to both. If `Carrier Selection/Entry` is an internal part only (not for direct use by designers), prefix it with `_` to signal that.

### 2d. Mixed hierarchy separators in set names

Some domain-scoped sets use a slash hierarchy (`Shipper/Tender/List Entry`, `Shipper/Contract/List Entry`, `Page/Header/Shipper`) while others use a dash-space pattern (`Shipper - Price Offers - Action Stripe`, `Tender Header - Shipper`). This makes autocomplete and scripted discovery inconsistent.

**Recommended standard:** Slash hierarchy for all domain groupings — e.g. `Shipper/Price Offers/Action Stripe`, `Shipper/Price Offers/Header`.

### 2e. `Check, Radio, Switch Item` vs. `Switch Item` — overlap

| Set | Count |
|---|---|
| `Check, Radio, Switch Item` | 12 |
| `Switch Item` | 8 |

These are likely the same conceptual component at different fidelity levels. The comma-list name is unusual and non-searchable. If they serve different use cases, rename both clearly; if one supersedes the other, deprecate the old set.

### Summary table

| Issue | Sets affected | Recommended standard |
|---|---|---|
| Case mismatch | `Route Entry`, `Route entry`, `Route Entry 2` | Consolidate into `Route Entry` |
| Singular/plural | `Status/Transports`, `Status/Requests` vs. `Status/Contract`, `Status/Tender` | All singular |
| Undocumented sub-set | `Carrier Selection/Entry` | Add description or `_` prefix |
| Mixed hierarchy style | `Shipper - Price Offers - Action Stripe`, `Tender Header - Shipper` | Slash hierarchy throughout |
| Duplicate/overlap | `Check, Radio, Switch Item` + `Switch Item` | Audit and consolidate |

## 3. Token Coverage

### 3a. Categories present

| Category | Source | Notes |
|---|---|---|
| Colors — raw ramps | styles + variables | Blue (11 stops), Neutral (13 stops), Red (2), Green (1), Yellow (1) |
| Colors — semantic | styles + variables | `text/*`, `background/*`, `border/*`, `button-*/`, `input/*`, `tabs/*`, `success/`, `warning/`, `danger/` |
| Typography | styles + variables | 24 named styles across `headline/h1–h6`, `body/md-*`, `sm/sm-*`, `xs/xs-*`, `lg/lg-*`, `Special/*` |
| Spacing | variables | `spacing/0`–`spacing/16` (12 stops, 4px base), `layout-gap-xs/sm/md/lg` |
| Border radius | variables | 8 stops: `0`, `2xs`, `xs`, `sm`, `md`, `lg`, `xl`, `full` — complete scale |
| Shadows | styles | `Default` (elevation), `left` (accent border) — only 2 values |
| Icon sizes | variables | 6 stops (10–24 px) |
| Component-level tokens | variables | `button/*`, `input/*`, `tab/*`, `container/*` |

### 3b. Gaps

**No focus/outline token.** Nothing named `focus`, `ring`, or `outline` exists in any category. Focus states for interactive components rely on ad-hoc values, and accessibility cannot be enforced systematically.

**No elevation scale.** Only two shadow styles: `Default` and `left`. A proper elevation ramp (`shadow/sm`, `/md`, `/lg`, `/overlay`) is absent. Dialogs, dropdowns, popovers, and tooltips presumably all share the same `Default` token, making layering order ambiguous.

**Spacing gap at the large end.** The scale jumps from `spacing/8 (32)` directly to `spacing/16 (64)`. There is no named token for the 32–64 px range, which is commonly needed for section-level padding.

**No dark-mode variable collection.** All 359 variables live in a single mode. If dark mode is a future requirement, no semantic alias layer is in place to support it.

### 3c. Token duplication and fragmentation

The single biggest token health issue is massive duplication of raw hex values under different names. Examples of the most-duplicated values:

| Hex | Name count | Representative names |
|---|---|---|
| `#FFFFFF` | 19 | `primary/text`, `background/body`, `neutral/0`, `neutral/white`, `base/White`, `container/background`, `body/background` |
| `#11151D` | 13 | `text/body`, `input/text`, `button-body/text`, `body/text`, `body/text-color`, `button-inverted/background` |
| `#105BFF` | 5 | `primary/background`, `background/primary`, `blue/500`, `button-primary/background`, `text/primary` |
| `#FFFFFF00` | 10 | `special/transparent`, `button-ghost/background`, `secondary/border`, `tabs/tab-default/border` |

This signals that no single-source-of-truth alias chain exists. Raw ramp tokens (`blue/500`, `neutral/0`) and semantic tokens (`background/primary`, `text/body-inverted`) are defined as independent flat values rather than the semantic layer referencing the ramp. A brand color change would require updating 5–19 entries instead of 1.

**Two parallel button token namespaces with conflicting values.** `button-primary/background` resolves to `#105BFF`; `button/primary/bg` resolves to `#0048E3`. These are different blues. Components binding one vs. the other render differently. This is a live defect.

**Orphan groups from a prior token schema.** `base/Primary (#0944DB)`, `base/Light`, `bg/Light`, `colors/black/*`, `Headlines/*`, `foreground/body (#101525)` duplicate values already present in the current schema under different names. They should be removed or replaced with aliases pointing to the canonical tokens.

## 4. Component Completeness

### 4a. Descriptions — zero coverage

**100% of component sets (57/57) have zero descriptions. All 741 individual component variants also have empty description fields.** The library is opaque to any consumer who is not already deeply familiar with it. Figma's component picker, code-connect outputs, and this repo's generated documentation all display blank fields.

### 4b. Variant depth — notable sets

| Set | Variants | Assessment |
|---|---|---|
| `Flag` | 271 | Comprehensive country-flag icon set — complete |
| `Button` | 27 | 9 variants × 3 states (Default/Hover/Disabled). Missing: `Focus` state on all variants |
| `Input` | 24 | 4 types × 2 Empty × 2 Disabled. Missing: `Error`/`Invalid` state, `Focus` state |
| `Carrier Offer` | 14 | Solid coverage across offer states and disabled flags |
| `Check, Radio, Switch Item` | 12 | Reasonable coverage but `Focus` state absent |
| `Illustration` | 12 | Present but no descriptions indicate when each illustration is appropriate |
| `Status` | 8 | Contains `Status=Status8` — a placeholder/unnamed status that must be resolved |
| `Badge` | 7 | Contains `Variant=Variant7` — unnamed placeholder, same issue |
| `Notification (Alert)` | 5 | Uses `Property 1=` as property name instead of a semantic label (e.g. `Variant=`) |
| `Table Cell` | 6 | Head + Cell × 3 alignments only. No `Sorted`, `Hover`, `Selected`, or `Checkbox` cell variants |
| `Dialog` | 1 | Single variant (`Headline=True`). No `Headline=False`, no loading/confirmation state |
| `Tabs` | 2 | Horizontal + Vertical orientations only. No individual `Tab Item` set, no `Disabled` tab state |
| `Pagination` | 2 | Only `First` and `Middle` variants. Missing: `Last`, `Single` (one page total), `Disabled` |
| `Status/Contract` | 4 | Contains `Status=Status8` — unnamed placeholder |

### 4c. Notable missing states across key components

- **Focus / active ring** — no component set exposes a `State=Focus` variant. This is a WCAG 2.4.7 (Focus Visible) concern.
- **Loading / skeleton** — no `State=Loading` variant on `Button`, `Table`, `Input`, or `Dialog`.
- **Input error state** — `Input` lacks `Error=True` variants. The `Form Control` standalone provides error styling but is not in a named set.
- **Empty state within Table** — `Table` appears only as a standalone component; no named set, no `Empty` state variant.

| `Accordion` | Not found in any set or standalone | Figma page `⚛️ Accordion` exists but no published component |

### 5b. Standalone components that are candidates for promotion to sets

The 188 standalone (null-set) components include many composite or reusable pieces that should be sets:

- `Page`, `Page/Footer`, `Page/Main Content Container` — layout scaffolding; candidates for a `Page/*` set
- `Alert Dialog` — distinct from `Dialog`; should be a named set
- `Table`, `Table Column`, `Route-Table` — table family should be consolidated into one set
- `Dropdown Menu`, `Popover` — parent containers for `Dropdown Item` / `Popover Item` sets
- `Form Control` — critical for form building; needs a set with Error, Hint, Required variant dimensions
- `Progress Bar/Label Description`, `Progress Group` — `Progress Bar/Bar` is a named set; companion pieces are not
- `Option Cards` (plural composite) and `Option Card` (singular set) exist separately; the relationship is uncodified
- `Proposals/List entry`, `Q&A/Entry`, `Chat list entry` — domain list items that would benefit from set grouping

### 5c. Icon duplication

Of the ~130 icon components in the standalone list, several appear **twice** under the same name (e.g. `bell`, `envelope`, `pen`, `truck`, `location-dot`, `circle-check`, `square-xmark`). These are likely accidental duplicates from different artboards. They should be deduplicated and consolidated into a single `Icon` set keyed by icon name.

## 6. Priority Actions

**P1 — Add descriptions to the 10 highest-traffic component sets** (impact: highest)  
`Button`, `Input`, `Badge`, `Table Cell`, `Notification (Alert)`, the `Status/*` family, `Dialog`, `Form Control`, `Carrier Offer`, `Dropdown Item`. Even a two-sentence description per set would unblock the `ds-component-describer` subagent and eliminate the single largest usability gap in the library. Currently 0 of 57 sets are documented.

**P2 — Resolve the duplicate button token namespace** (impact: live defect)  
`button-primary/background (#105BFF)` and `button/primary/bg (#0048E3)` point to different hex values. This causes inconsistent button rendering depending on which token a component binds. Audit all `button/*` vs. `button-primary/*` token groups, designate one canonical set, and remove or alias the other.

**P3 — Consolidate `Route Entry` / `Route entry` / `Route Entry 2`** (impact: naming + discoverability)  
Three sets with overlapping purpose and a casing inconsistency. Designers using the component picker cannot tell which to use. Merge into one `Route Entry` set. This also improves naming convention scores for any tooling that checks consistency.

**P4 — Add `Error` and `Focus` states to `Input` and `Button`** (impact: completeness + accessibility)  
`Input` has no `Error=True` variant; neither `Input` nor `Button` has a `State=Focus` variant. Form validation error display and keyboard-navigation focus rings cannot be designed consistently without these variants in the library. This is also a prerequisite for WCAG 2.4.7 compliance.

**P5 — Introduce a semantic alias layer for raw color tokens** (impact: maintainability + future-proofing)  
The current flat token structure stores the same hex under 5–19 different names. Establish primitive tokens (`blue/500`, `neutral/0`, etc.) as the single source of truth, then have semantic tokens (`background/primary`, `text/body-inverted`, etc.) reference them by alias rather than by hardcoded hex. This is a foundational change that prevents color drift and is a prerequisite for any theming or dark-mode work.
